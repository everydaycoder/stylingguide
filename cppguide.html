C++ Styling Guide

C++ is a language that is very powerful, and therefore, in some cases, superior. This power comes in the form of pointers, raw assembly instructions, memory management, you name it. But these also aren't necessarily the prettiest things to look at when implemented in code.

Code that involves these kinds of things becomes hard to manage and maintain, and so nobody can get any work done at the end of the day. Since all of the documentation and code our developers write get shared with others, this is a big problem. So what we did is come up with a set of guidelines you should follow in order to make your code as readable as possible.

This document also is meant to be something you have to study for days on end in order to perfect. We've compacted it into such a small size, that you could learn almost everything there is to know here in a few hours, if not less than an hour. This is a result of the strive to make the Google Style Guides easier to abide by, since these documents sometimes take even a few weeks to perfect.

Headers

Include Guard

In a program with multiple files, which, assuming you do, since this is a header file we're talking about, you may include the same header file more than once. This can lead to absolutely catastrophic errors, and ones that absolutely cannot be recovered from. To fix this, we need a way of making sure that only one instance of a header ever exists during runtime.

Since this is a compiler error, we need to use preprocessing code. Look at the code below to see what this implementation may look like.

#ifndef __FOO_BAR_BAZ_H__
#define __FOO_BAR_BAZ_H__

// ... declare things here

#endif // __FOO_BAR_BAZ_H__ (See Closing Comments)
An example of an apparently very bad naming system. Links: Closing Comments

As you can see, we check if a macro that was specifically tailored as an id for the header file exists, and if it doesn't, we define it. Note the fact that we don't have to give it a value. In fact, at a large scale, you REALLY don't want to define a value, because it saves space. Don't forget to end the if statement at the end of the file, or else it will be an error in and of itself.

The naming convention is nice and simple. All that this convention assumes is that you have your header file in a directory that isn't named exactly after the project name, disregarding case.

1
2
#ifndef __PROJECT_PATH_NAME_H__
// ... Other parts of the file
Shows the naming convention of the Include Guard, where PROJECT is the project name, PATH is the path to the file*, and NAME is the name of your header file, obviously without the `.h' part.

* The path should really only be the file's parent directory, but in the case you need a full path, slashes are replaced with underscores.
